# Автоматизация ретроспективного клинического исследования

Воспроизводимый пайплайн на примере исследования антибиотикопрофилактики
ИМВП при внутридетрузорном введении БТА. Данные **синтетические**, но
структурно идентичны реальному «грязному» Excel: даты как Excel serial
numbers, возраст внутри ФИО, свободный текст в микробиологии, дубликаты
процедур между «двумя таблицами», опечатки в годах.

Этот проект — не про конкретные числа, а про **процесс**: как превратить
разовый ad-hoc анализ в инженерно-воспроизводимое исследование.

---

## TL;DR — запуск

```bash
pip install -r requirements.txt
python run.py --generate     # 1) сгенерировать синтетические raw-данные
python run.py                # 2) прогнать весь анализ
python -m pytest -q          # 3) проверить трансформации
# интерактивно:
jupyter lab notebooks/analysis.ipynb
```

## Три способа организовать такое исследование (плюсы/минусы)

Главный вопрос — **где живёт логика**. От этого зависит воспроизводимость.

### A. Чистый пайплайн (скрипты/модули, без ноутбука)

| Плюсы | Минусы |
|---|---|
| Идемпотентность: один вход → один выход | Высокий порог входа, много «лесов» |
| Тестируемость каждой трансформации | Слабая интерактивность для разведки (EDA) |
| Версионируется, ложится в CI | Промежуточные результаты не «потрогать» глазами |
| Единый источник чисел | Избыточен для разового вопроса |

Когда брать: данные обновляются, анализ повторяется, результат идёт в публикацию/регуляторику.

### B. Чистый Jupyter-ноутбук

| Плюсы | Минусы |
|---|---|
| Мгновенная обратная связь, отличная EDA | **Скрытое состояние** и выполнение ячеек не по порядку |
| Код + вывод + нарратив в одном месте | Плохие диффы, копипаст-дрейф логики |
| Низкое трение для прототипа | Почти не тестируется, не идемпотентен |

Когда брать: разовая разведка, обучение, демонстрация.

> Именно режим B породил в реальном кейсе цикл «нашли артефакт → поправили
> руками → пересчитали весь анализ → числа разъехались». Это не вина
> аналитика — это свойство среды со скрытым состоянием.

### C. Гибрид — детерминированное ядро + тонкий ноутбук *(рекомендуется)*

Ядро (`src/bta_study`) — протестированная библиотека чистых функций.
Ноутбук импортирует её и делает только разведку/визуализацию/нарратив.

| Плюсы | Минусы |
|---|---|
| Логика протестирована и идемпотентна | Нужна дисциплина: что в ядро, что в ноутбук |
| Ноутбук = тонкая презентация поверх истины | Небольшой стартовый оверхед |
| Отчёт, слайды, ноутбук берут числа из ОДНОГО объекта | |
| EDA сохраняется, но не определяет результат | |

Для клинического исследования это правильный дефолт, и проект собран именно так.

## Общая методика и алгоритм (что автоматизируем)

Любое исследование на real-world data проходит одни и те же этапы. В реальном
чате они были перемешаны; здесь — разделены и у каждого один источник истины.

```
0. ПРОТОКОЛ            config/protocol.yaml   ← клинические решения, единица анализа
        │                                       конечные точки, пороги
        ▼
1. INGESTION          src/.../ingest.py       ← «две таблицы» → одна длинная
        │
        ▼
2. ПАРСИНГ             src/.../parsers.py      ← чистые функции (даты, возраст,
        │                                        микробиология, антибиотики) + ТЕСТЫ
        ▼
3. FEATURES           src/.../features.py      ← интервалы, конечные точки
        │
        ▼
4. ДЕДУПЛИКАЦИЯ        src/.../dedup.py        ← ключ (пациент, дата БТА), идемпотентно
        │
        ▼
5. ВАЛИДАЦИЯ           src/.../schema.py       ← Pandera-контракт; падает на артефакте
        │
        ▼
6. СТАТИСТИКА          src/.../stats.py        ← Fisher, McNemar, Wilcoxon, Bayes
        │
        ▼
7. РЕЗУЛЬТАТЫ          pipeline.Results        ← единый объект → отчёт/слайды/ноутбук
```

**Ключевой инвариант:** число процедур в анализе
= сырые строки − дубликаты − исключённые по возрасту. Этот баланс
проверяется тестом и печатается как «происхождение чисел» (provenance),
чтобы любой результат можно было проследить до входа.

## Какие инструменты и зачем

| Слой | Инструмент | Что решает |
|---|---|---|
| Парсинг дат | свой `parse_date` + распознавание Excel serial | serial numbers vs строки |
| Валидация | **Pandera** | контракт на данные, ловит артефакты до анализа |
| Малые выборки | **statsmodels** (McNemar, proportion CI), **scipy** (Fisher, Wilcoxon) | парные тесты, точные интервалы |
| Неопределённость | **Beta-Binomial (scipy)** | честная оценка доли при 0–3 событиях |
| Воспроизводимость | фиксированный `seed`, идемпотентные функции, тесты | один вход → один выход |
| Конфигурация | **YAML-протокол** | клинические решения вне кода |
| Тесты | **pytest** | каждый артефакт закрыт тестом навсегда |
| Гибрид | **jupytext** (`.py` ↔ `.ipynb`) | ноутбук диффится как код |

**Что ещё можно добавить под рост проекта:**
- `lifelines` — time-to-event, если окно наблюдения переменное (цензура);
- `PyMC` — иерархическая байес-модель (пациент как случайный эффект для
  повторных процедур);
- оркестратор `snakemake`/`prefect` — если этапов станет много и нужен кэш;
- `DVC` — версионирование самих данных вместе с кодом;
- `Quarto` — единый источник для отчёта и слайдов из того же `Results`.

---

## Почему именно эти статистические методы

- **McNemar (парный)** — для ОАМ vs посев на одних процедурах. Обычный
  χ²/Fisher здесь некорректен: тесты не независимы, это одни и те же пациенты.
- **Wilcoxon signed-rank** — сроки ОАМ vs посев: парные, распределение
  ненормальное.
- **Clopper-Pearson / Wilson** — точные интервалы доли вместо нормального
  приближения, которое при малых n врёт.
- **Beta-Binomial** — апостериорная доля осмысленна даже при k=0; даёт
  интерпретируемый интервал вместо «p=0.098, недостоверно».
- **Fisher exact** — факторы vs исход на малых таблицах, но с явным
  предупреждением о мощности: при ≤5 событиях незначимость ≠ отсутствие эффекта.

## Что осталось ручным (и почему так правильно)

Автоматизируется всё *детерминированное*. Ручными остаются только **клинические
решения** — но они вынесены в `protocol.yaml` явно, с обоснованием:
- единица анализа (процедура vs пациент) — даёт расхождение 64 vs 66;
- трактовка сомнительного случая (субфебрилитет считать ИМВП?) —
  политика `ambiguous_uti_policy: sensitivity` (считать оба варианта);
- порог значимого титра, окно «свежести» теста.

Меняешь значение в YAML — весь анализ перепроливается, без правок в коде.

## Структура проекта

```
bta_study/
├── run.py                      # CLI: --generate / анализ
├── config/protocol.yaml        # ВСЕ клинические решения здесь
├── src/bta_study/
│   ├── generate.py             # генератор грязных синтетических данных
│   ├── ingest.py               # «две таблицы» → одна
│   ├── parsers.py              # чистые тестируемые парсеры
│   ├── features.py             # производные признаки и конечные точки
│   ├── dedup.py                # дедупликация процедур
│   ├── schema.py               # Pandera-контракт
│   ├── stats.py                # вся статистика
│   └── pipeline.py             # оркестрация → Results
├── notebooks/analysis.py       # тонкий слой (percent-format, ↔ .ipynb)
├── tests/                      # pytest: парсеры, дедуп, пайплайн
└── data/{raw,processed}/       # вход/выход (raw генерируется)
```
